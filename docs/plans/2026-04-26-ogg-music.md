# Plan: OGG-streamed music for sx-doom-overlay

**Branch:** `feat/task-9b-ogg-music` (off `main` @ `9cb4785`)
**Status:** PENDING
**Type:** Feature
**Worktree:** No
**Approved:** Yes (user greenlit 2026-04-26)

## Why

Real-time MIDI synthesis through Nuked-OPL3 (the path on `feat/task-9-music`)
sounds inherently FM-era: nasal, thin on bass, dynamic-range-narrow. Doom's
music was *composed* on Roland SC-55 wavetable hardware; OPL was always a
"good enough" DOS fallback. Comparing OPL emulation to "the real OST" is
comparing FM to wavetable — FM always loses that fight.

Pre-rendering each MIDI lump offline with FluidSynth + a high-quality GM
soundfont, then streaming the resulting OGG file at runtime, gives:

- Authentic SC-55-quality sound (the OST users remember)
- Constant, low CPU cost during gameplay (~5% of one core for libvorbis)
- No risk of music-induced lag, regardless of voice count or MIDI complexity
- Frees ~500 KB heap (OPL stack ~700 KB → stb_vorbis state ~200 KB)
- Drops 7 K lines of vendored chocolate-doom OPL code

Storage cost is on the SD card (~150 MB for full Doom 1 OST), where SD cards
have effectively infinite room. User runs a one-time render script on
desktop to produce the OGG pack from their own IWAD.

## Architecture

Music subsystem becomes a file streamer instead of a MIDI synthesizer.

```
engine                                                       hardware
──────                                                       ────────
S_StartMusic("d_e1m1")                                            │
   │                                                              │
   ▼                                                              │
music_module_t::RegisterSong(buf, len)                            │
   │  · buf is the MUS lump from the IWAD                         │
   │  · we extract its 8-char lump name                           │
   │  · resolve to /switch/sx-doom-overlay/music/<name>.ogg       │
   │  · open file, init stb_vorbis pushdata decoder               │
   │  · read LOOPSTART/LOOPEND vorbis comments                    │
   ▼                                                              │
music_module_t::PlaySong(handle, looping)                         │
   │                                                              │
   ▼                                                              │
[audio submit thread, every buffer]                               │
   │  music_render(scratch[FRAMES_PER_BUF * 2])                   │
   │  · stb_vorbis_get_samples_short_interleaved → scratch        │
   │  · on EOF: if looping seek to LOOPSTART, else stop           │
   │  · scale by current_music_volume / 127                       │
   ▼                                                              │
bus_mix(sfx, music) → audoutAppendAudioOutBuffer ────────────────►│
```

OGG files are opened lazily on `RegisterSong`. Decode happens on the audio
submit thread (same thread that used to drive `OPL_LIBNX_Render`). No new
threads, no extra mutexes — replaces the OPL render call 1:1.

## Risks

- **stb_vorbis decode latency at 48 kHz stereo on Tegra X1.** Empirically
  ~1-2 ms per 1100-frame buffer. Well under the 50 ms buffer budget.
- **SD card read jitter.** stb_vorbis pushdata API consumes bytes
  on demand; if read stalls, music underruns briefly. Mitigation:
  pre-decode N seconds into a ring buffer ahead of `music_render`.
- **Loop point glitches.** Seeking back to LOOPSTART mid-frame can cause
  a click. Mitigation: crossfade ~10 ms across the loop seam.
- **Heap.** stb_vorbis pushdata setup ~150 KB. Net free vs OPL: +500 KB.
  Comfortable margin.
- **Asset distribution.** Users without the render script can't get music.
  Mitigation: clear README + a sample test.ogg shipped with the release zip
  so the path works on first launch (proves audio chain works) before they
  commit to rendering their own pack.

## Success criteria

- E1M1 plays its proper "At Doom's Gate" track at SC-55 quality on hardware
- Track switches cleanly between maps
- No music-induced framerate drops or audio underruns over 10 minutes of play
- OPL stack removed from build; binary smaller; heap free higher than current
